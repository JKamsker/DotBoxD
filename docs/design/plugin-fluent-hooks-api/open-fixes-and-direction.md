# Plugin context & hooks: open fixes and design direction

Companion to [server-walkthrough.md](server-walkthrough.md),
[../remote-plugin-server-builder/interface-driven-plugin-server.md](../remote-plugin-server-builder/interface-driven-plugin-server.md),
and [kernel-binding-model.md](kernel-binding-model.md).

**Status:** Design direction + review backlog for PR #88 (`codex/improve-hooks-issue-87`, issue #87).
**Date:** 2026-06-23.

Design guide this doc is measured against: **Simple · Obvious · Discoverable · Consistent · Minimal · Composable.**

---

## TL;DR

PR #88 ("plugin-owned hook contexts") moves the fluent API in the right direction — the unified
`(value, ctx)` delegate shape and the `On<TEvent>()`-defaults-to-a-typed-context ergonomics are real
wins. But it is **not mergeable as-is** (two semantic P1s + red CI), and more importantly its context
mechanism is the **one part of the plugin-server facade that is not contract-driven**. Everything else
generated by `[GeneratePluginServer]` is derived from a declared interface; the context is derived from a
string naming convention. That single inconsistency is the root of the worst findings and the gap between
"magical" and "obvious."

The direction: **make the context a declared contract like the rest of the surface already is**, and
**describe a host capability once** so the lowered-IR call site and the IPC call site stop being two
hand-synced string contracts.

---

## Part 1 — Where we already are (the good news)

The "server declares how it wants to be talked to; the client writes one line; codegen fills the rest"
model **already exists and is genuinely contract-driven** for every part of the facade *except* the
context.

The author writes one line:

```csharp
[GeneratePluginServer]
public partial class GamePluginServer : IGameWorldAccess;
```

From the `[DotBoxDService]` interface graph the generator **enumerates members** and emits the whole
facade — RPC proxy forwarders, the `IPluginServer<IGameWorldAccess>` lifecycle, the `Setup` accumulator,
live-settings `Get`, `IGameWorldServer`, and `GamePluginServerBuilder`:

| Generated artifact | Driven by | Where |
|---|---|---|
| World resolution | the interface carrying `[DotBoxDService]` (not the class name) | [PluginServerFacadeModelFactory.cs:71](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerFacadeModelFactory.cs) `ResolveWorldType` |
| Controls / forwarders / scoped clients | walking the interface's members + nested `[DotBoxDService]` returns | [PluginServerFacadeModelFactory.cs](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerFacadeModelFactory.cs) `ResolveControls` (:89), `ResolveMethods` (:125), `ResolveReturnWrapper` (:196) |
| `I{World}Server`, lifecycle, builder | the world **type symbol** | [PluginServerFacadeEmitter.cs:20](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerFacadeEmitter.cs), [PluginServerFacadeNameFormatter.cs:30](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerFacadeNameFormatter.cs) |
| **Context + hook/sub registries** | **the class name string** (`FacadeRootName(name) + "Context"`) | [PluginServerContextSurfaceEmitter.cs:16](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerContextSurfaceEmitter.cs), [PluginServerFacadeNameFormatter.cs:21](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/PluginServer/PluginServerFacadeNameFormatter.cs) |

The server-side contract (`[DotBoxDService] interface IGameWorldAccess`, its DTOs, and the
`[HostCapability]`/`[HostBinding]` annotations) lives in a separate `Server.Abstractions` assembly. The
server owns the contract; the plugin writes one line; the generator does the rest. **This is exactly the
model you described — it already ships.**

The lone exception is the context. It is named by string convention, **enumerates no contract**, and the
author extends a magically-named `partial class GamePluginContext` with free-form members
([GamePluginContext.cs:3](../../../samples/GameServer/Examples.GameServer.Plugin/GamePluginContext.cs)).
Nothing declares the context's intended surface, so *what is generated vs hand-written is invisible and
unchecked* — the opposite of Obvious/Discoverable, and the precise reason its identity inference is
fragile (see P2 below).

## Part 2 — Is the "RPC pipeline" one thing or two? (the unification question)

**Answer: ~80% one thing by design, but the seam is a hand-maintained string contract, not a single
source of truth.** There are three runtime paths; two already converge, one is correctly separate.

1. **Host call inside lowered IR** — a `Where`/`Select`/`Run` (or a server-extension body) that touches a
   host member is lowered to a sandbox `CallExpression(bindingId, args)`, where
   `bindingId = host.{ns}.{Type}.{Method}`
   ([DotBoxDHostBindingExpressionLowerer.cs:89](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/Lowering/Expressions/DotBoxDHostBindingExpressionLowerer.cs)).
   At runtime the interpreter resolves it through `BindingRegistry.GetDescriptor(id)` and the host runs the
   real C# method **server-side**
   ([ExpressionEvaluator.Calls.cs:149](../../../src/Kernels/DotBoxD.Kernels.Interpreter/Internal/Expressions/ExpressionEvaluator.Calls.cs),
   [BindingRegistry.cs:44](../../../src/Kernels/DotBoxD.Kernels/Bindings/BindingRegistry.cs)).

2. **Server-extension / `InvokeAsync` request-response** — a whole verified function installed as a kernel,
   reached by `InstalledKernel.InvokeServerExtensionAsync` and dispatched **by `pluginId`**.
   `InvokeAsync` is literally an *anonymous* server extension: it reuses the same `DotBoxDRpcJsonLowerer`
   and synthesizes `"rpcEntrypoint":"Invoke"`
   ([InvokeAsyncModelFactory.cs](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/InvokeAsync/InvokeAsyncModelFactory.cs)).
   The in-process `ServerExtensionProxy` and the over-IPC interceptor **terminate at the same
   `InvokeServerExtensionAsync` entrypoint — only the transport differs.**

   These two converge: a server-extension body is lowered through the *same* host-binding lowerer, so
   Layer-1 host calls are emitted *inside* the Layer-2 entrypoint. "Used in a filter lowered to IR" and
   "called over IPC" already meet at `InvokeServerExtensionAsync`. The shipped `[ServerExtension]`
   mechanism is the cleanest proof: one method body lowered to a verified-IR kernel, reachable **both**
   in-process (`ServerExtensionProxy`) **and** over IPC (a generated client proxy,
   [RpcKernelClientProxyEmitter.cs](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/Rpc/Client/RpcKernelClientProxyEmitter.cs)).
   The *proposed* `[KernelRpcService]` follow-up ([followups.md](followups.md) §2) would put a **declared
   interface** in front of exactly this machinery — which is precisely the contract-first direction §3
   argues for.

3. **`RunLocal` / `RegisterLocal` IPC terminal** — a *third*, genuinely separate mechanism: push-only,
   keyed by `subscriptionId`, running a **native delegate in the plugin process**. It shares only the
   value marshaller; it never reaches the binding registry
   ([RemoteLocalHandlerRegistry.cs](../../../src/Hosting/DotBoxD.Plugins/Runtime/Hooks/RemoteLocalHandlerRegistry.cs)).
   This one *should* stay distinct — it is the "cross the wire once, then run arbitrary native code"
   terminal, by design.

So from the server dev's POV it largely *is* the same thing glued with codegen — with one real wart:

> The binding id (`host.{ns}.{Type}.{Method}`) and the allocation/effect classification are **re-derived
> independently** in the analyzer ([DotBoxDHostBindingExpressionLowerer.cs](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/Lowering/Expressions/DotBoxDHostBindingExpressionLowerer.cs),
> `DotBoxDManifestEffectModel`) and in the runtime (`HostServiceBindingFactory`), with code comments
> literally begging the two to stay in sync ("Must match HostServiceBindingFactory.ReturnAllocates").
> Drift fails at **install** (`DBXK041`), not at compile time. That is the "wire it twice" smell.

**Unification target:** describe a host capability **once** (one attributed interface method) and *project*
it into both dispatch tables from a single shared descriptor, instead of two assemblies mirroring the same
string/effect logic. Then there are exactly two runtime concepts — resolve-by-binding-id (host calls) and
resolve-by-pluginId (whole entrypoints) — over one transport abstraction, plus the deliberately-separate
native `RunLocal` terminal.

## Part 3 — The design direction

### 3.1 Make the context a declared contract (the core move)

Bring the context under the same contract-driven model as the world surface. The author (server or plugin)
**declares** the context's shape — an interface or abstract type, the same way `IGameWorldAccess` is
declared — and the generator **enumerates its members** (reusing `ResolveControls`/`ResolveMethods`) to
emit the typed context surface. The context stops being a convention-named partial you graft onto blind.

Benefits, by principle:

- **Discoverable / Obvious** — the context's surface is *declared in one place* you can grep and read,
  not discovered by typing members into a magic partial. What's generated vs hand-written becomes visible.
- **Consistent** — the context derives from an enumerable contract + attributes, exactly like every other
  generated member. One discovery mechanism, not two.
- **Composable** — feed the existing interface-enumeration machinery; the generator stays small.
- **Minimal** — *keep the one-liner*. The convention default (parameterless `On<TEvent>()` binding the
  plugin's context) is good and stays; only the context's **shape** gains a contract. No call-site ceremony.

### 3.2 Make the three execution tiers obvious in the contract

A context member can run in one of three places. Today the only signal is an attribute on a free-form
partial, which is easy to miss. Make the distinction visible on the declared contract:

| Tier | Marker | Runs | Usable in |
|---|---|---|---|
| Pure helper, inlined into IR | `[KernelMethod]` | server-side sandbox | lowered `Where`/`Select`/`Run`/`Register` |
| Host capability | `[HostBinding(id, capability, effects)]` | server-side host | lowered stages **and** server extensions |
| Native | (none) | plugin process, post-IPC | `RunLocal`/`RegisterLocal` only |

Consider symmetric `…Local` verb pairs so the **verb names the execution site** and the lambda body is
identical either way (`Run` = lowered server-side, `RunLocal` = native in-process). That is the Orleans
"location-transparent contract" idea applied to terminals, and it answers the "is it the same code?"
question with: *yes, literally the same expression — the suffix chooses where it runs.*

### 3.3 Fix identity so inference is local and owned, not global and heuristic

Three of the worst findings share one root cause — **identity inferred by name/global scan instead of by
ownership**. Resolve the context from the owning server symbol (greppable, like the
`[GeneratePluginServer]` attribute already is), not from a whole-compilation scan that silently bails when
it finds more than one server. This directly removes P2 (below) and makes multi-server projects work.

### 3.4 Collapse the host-capability seam to one descriptor

Per Part 2: emit a single capability descriptor consumed by **both** the analyzer lowering and the runtime
binding factory, so `host.{ns}.{Type}.{Method}` and the effect/allocation classification are defined once.
Removes the `DBXK041`-at-install drift class entirely.

## Part 4 — Open fixes (review backlog)

Ranked. P1 = blocks merge; P2 = fix before this is "the library shape"; P3/mechanical = polish. Each is
verified against the branch.

### P1 — blockers

1. **Explicit context factories are silently collapsed by the `(TEvent, TContext)` cache key.**
   On a cache hit the registry returns the existing pipeline and **discards the new `createContext`** (it
   is null-checked, then thrown away). Two helpers that each set up `On<E, Ctx>(factory)` silently share
   the first factory → handlers run with the wrong context; composition is order-dependent.
   [HookRegistry.cs:86](../../../src/Hosting/DotBoxD.Plugins/Runtime/HookRegistry.cs),
   [SubscriptionRegistry.cs:59](../../../src/Hosting/DotBoxD.Plugins/Runtime/Subscriptions/SubscriptionRegistry.cs).
   *Fix:* throw/diagnose on a conflicting factory for an existing key, or key on the factory identity.
   _(Composable, Obvious.)_

2. **Result-hook priority is no longer global once an event has multiple context pipelines.**
   `FireManyAsync` walks pipelines in dictionary order and returns the first non-null result; priority is
   only sorted *within* one pipeline. A priority-`0` handler in an earlier-created context pipeline beats a
   priority-`100` handler in a later one. This PR is what makes multiple-pipelines-per-event reachable, so
   it introduces the regression.
   [HookRegistry.Pipelines.cs:59](../../../src/Hosting/DotBoxD.Plugins/Runtime/HookRegistry.Pipelines.cs),
   [ResultHookSlot.cs:240](../../../src/Hosting/DotBoxD.Plugins/Runtime/Hooks/ResultHookSlot.cs).
   *Fix:* merge-sort entries across pipelines by global `(priority, order)` before dispatch. _(Obvious, correctness.)_

3. **CI is red on a live-run failure.** The generated-package unit tests pass (5/5) and the sample builds
   clean, but the Windows `Build` job's GameServer **smoke run** throws `RemoteServiceException: Internal
   error` from `BlinkBehindAsync` at runtime. Not reproduced locally, so possibly environment/timing — but
   it is a *runtime* server-extension dispatch fault, not a codegen test, and must be green (and
   root-caused, not just rerun) before merge.

### P2 — design hazards

4. **Generated-remote fallback is too broad.** `CandidateKind` claims any receiver member literally named
   `Hooks`/`Subscriptions` with no ownership check in the fast path — a foreign fluent API using those
   names gets its call sites lowered.
   [GeneratedRemoteHookChainFallback.cs:23](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/HookChains/Support/GeneratedRemoteHookChainFallback.cs). _(Composable.)_

5. **Default-context inference is global and bails on ambiguity.** It scans every syntax tree for
   `[GeneratePluginServer]` and `return null`s the moment it finds a second distinct one — silently
   breaking the documented auto-context promise in any multi-server project.
   [GeneratedRemoteHookChainFallback.GeneratedContexts.cs:34](../../../src/CodeGeneration/DotBoxD.Plugins.Analyzer/Analysis/HookChains/Support/GeneratedRemoteHookChainFallback.GeneratedContexts.cs).
   Fixed by §3.3 (resolve by ownership). _(Composable, Obvious.)_

6. **The analyzer↔runtime host-binding seam is duplicated** (Part 2). `host.{ns}.{Type}.{Method}` + effect
   classification re-derived in two assemblies; drift fails at install (`DBXK041`). Fixed by §3.4. _(Consistent, Minimal.)_

### P3 / mechanical

7. **Generator plumbing leaks into IntelliSense** — `UseGeneratedLocalChain` / `UseGenerated*` /
   `UseProjecting*` crowd the author surface.
   [RemoteHookPipeline.cs:36](../../../src/Hosting/DotBoxD.Plugins/Runtime/Hooks/RemoteHookPipeline.cs),
   [RemoteHookStage.cs:34](../../../src/Hosting/DotBoxD.Plugins/Runtime/Hooks/RemoteHookStage.cs).
   *Fix:* `[EditorBrowsable(EditorBrowsableState.Never)]` + "generator-only" XML doc. _(Minimal, Discoverable.)_

8. **Public surface doubling** — every pipeline/stage now exists in both `<TEvent>` and `<TEvent,TContext>`
   forms (api-baseline +268/−60; the pipeline/stage type families roughly double) via `new`-shadowing
   return-narrowing shims. Defensible
   back-compat tax; decide consciously whether the `<TEvent>` shim layer survives long-term.
   [HookPipeline.Default.cs](../../../src/Hosting/DotBoxD.Plugins/Runtime/Hooks/HookPipeline.Default.cs). _(Minimal.)_

9. **Stale doc names** — `server.Kernels.Register` ([server-walkthrough.md:263](server-walkthrough.md)) and
   `server.Events` (should be `server.Subscriptions`, [:269](server-walkthrough.md)), inconsistent with the
   rest of the same doc. _(Discoverable.)_

10. **Dead code** — `IPluginEventPipelineRegistry` (one occurrence repo-wide, its own declaration).
    [ServerContextFactory.cs:17](../../../src/Hosting/DotBoxD.Plugins/Runtime/ServerContextFactory.cs). Delete (YAGNI). _(Minimal.)_

11. **Hook/subscription overload drift** — `RemoteSubscriptionPipeline.Typed` is missing the two
    element-only `UseGeneratedLocalChain` overloads `RemoteHookPipeline.Typed` has (parity regression vs
    `main`); ~90% copy-paste between the `Remote*.Typed` and `*Stage.Default` pairs invites exactly this.
    *Fix:* collapse onto a `kind`-parameterized base (mirror the generator's own
    `GeneratedRemoteHookChainKind`). _(Consistent, Minimal.)_

12. **Sample teaches noise** — `Program.cs:91` / `LocalReactions.cs` write `(e, _) =>` for filters that
    ignore the context; use `e =>` and reserve `(e, ctx) =>` for stages that touch `ctx`. _(Minimal.)_

13. **`RegisterLocal` has three authoring shapes** (the legacy cancellation `(e, ctx, ct) =>
    ValueTask<TResult>`), vs the "exactly two" rule. Drop it (`ctx.CancellationToken` exists) or document it
    as the deliberate async exception. _(Consistent.)_

## Part 5 — Suggested sequencing

1. **Unblock merge:** fix P1.1 (factory collapse) and P1.2 (global priority), root-cause P1.3 (CI). These
   are localized and don't depend on the redesign.
2. **De-risk identity:** §3.3 — resolve the context by ownership, which also closes P2.4 and P2.5.
3. **Then the bigger moves, separately:** §3.1 (context-as-contract) and §3.4 (single capability
   descriptor) are the design-shaping changes; land them as their own PRs, not bolted onto #88.
4. **Polish:** the P3/mechanical list, ideally alongside the duplication collapse (P3.11) so the
   hook/subscription axis can't drift again.

The core idea in #88 is worth keeping. The work is to make the context obey the contract-driven pattern the
rest of the facade already follows, and to make the one-source-of-truth seam real — turning the current
"magical" into "obvious, by codegen."
