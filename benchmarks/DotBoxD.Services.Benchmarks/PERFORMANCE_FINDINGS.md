# DotBoxD Services Performance Findings

All numbers are local Release stopwatch probes on the same machine and are intended
to keep optimization claims concrete.

| Finding | Probe | Workload | Before Total | Before ns/op | Before Alloc | Before B/op | After Total | After ns/op | After Alloc | After B/op | Notes |
| --- | --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| Listener-free server dispatch still allocated and populated a request telemetry scope. | `RpcTelemetryBenchmarks.SuccessfulRequestWithoutListeners` | Short BenchmarkDotNet run of successful request scopes with no activity or metric listeners | n/a | 55.49 | n/a | 64.0 | n/a | 0.54 | n/a | 0.0 | Detailed state is now created only when the duration histogram or activity source is enabled. The end-to-end `MovePlayerAsync` control confirmed the same 64 B/request reduction in both profiles (640 B to 576 B; 320 B to 256 B). Direct latency improved by 99.0%; asynchronous round-trip latency was noisy, so no end-to-end latency claim is made. |
| Repeated `RpcPeer.Get<TService>()` calls created a fresh generated root proxy. | `--probe-peer-proxy-cache` | 1,000,000 locked legacy proxy creations vs cached `RpcPeer.Get<IGameService>` calls | 33.5 ms | 33.5 | 32,000,040 B | 32.0 | 36.9 ms | 36.9 | 40 B | ~0 | Allocation-only improvement for the small root proxy in this probe; cache entries are invalidated by generated-registry registration stamps. |
| Generated `Get*` extensions bypassed the root proxy cache and directly constructed a new proxy. | `--probe-peer-proxy-cache` | 1,000,000 generated `GetGameService` calls, legacy direct constructor vs generated cached extension | 8.6 ms | 8.6 | 32,000,040 B | 32.0 | 37.9 ms | 37.9 | 40 B | ~0 | Allocation-only improvement with a latency tradeoff for repeated lookup-only calls; generated extensions now share the same cached root proxy identity as `RpcPeer.Get<TService>()`. |
| Owned `StreamConnection.ReceiveAsync` paid active-receive interlocked operations that only protect non-owned stream disposal. | `--probe-stream-connection-receive-tracking` | 1,000,000 owned empty-stream receives, with legacy tracking simulated around the current receive body | 124.2 ms | 124.2 | 72,001,584 B | 72.0 | 113.8 ms | 113.8 | 72,000,040 B | 72.0 | Time-only improvement; non-owned streams still track active receives so blocked reads are disposed during peer shutdown. |
| Stream and TCP frame reads wrapped existing `ValueTask<int>` operations in two private `Task<int>` state machines. | `--probe-stream-connection-receive-tracking`, `--profile-ipc namedpipe 2000 --low-alloc --no-timeout` | 1,000,000 synchronous 1 KiB ValueTask frame receives with finite / infinite idle timeout; 2,000 named-pipe Add / Echo roundtrips | 485.6 / 138.8 ms medians | 485.6 / 138.8 | 176,000,000 B | 176.0 | 433.9 / 140.0 ms medians | 433.9 / 140.0 | 32,000,000 B | 32.0 | Allocation claim: direct ValueTask forwarding removes exactly 144 B from the synchronous frame path; finite-timeout latency improves 10.6%, while infinite-timeout latency is unchanged/noisy. The mirrored TCP path uses the same single-consumption pattern. Controlled named-pipe runs fall from `2,217.6 / 2,265.1` to `1,673.7 / 1,720.6 B/call` for Add/Echo because pending prefix/body reads on both peers also avoid wrapper tasks. Legacy `ReceiveAsync` keeps its required Task adapter. |
| The primary frame-channel receive path wrapped each pending payload receive in another async `RpcFrame` state machine. | `--profile-ipc namedpipe 20000 --low-alloc --no-timeout` | 20,000 named-pipe Add / Echo roundtrips | n/a | n/a | 33,443,472-33,445,280 B / 34,401,168-34,401,536 B | 1,672.2-1,672.3 / 1,720.1 | n/a | n/a | 28,003,192-28,004,208 B / 28,960,568-28,962,208 B | 1,400.2 / 1,448.0-1,448.1 | Making `ReceiveFrameValueAsync` the native Stream/TCP core removes about 272 B per unary roundtrip. Legacy payload APIs detach through adapters, with a synchronous fast path for memory-backed reads; EOF ownership, timeout/cancellation, and concurrent-receive behavior remain covered. |
| Uncontended sends with a live caller token created a linked cancellation source before acquiring the free send gate. | `--probe-stream-connection-send-gate` | 1,000,000 synchronous valid-frame sends to a zero-allocation sink | 138.9-140.0 ms | 138.9-140.0 | 80,000,000 B | 80.0 | 96.9-102.5 ms | 96.9-102.5 | 0 B | 0.0 | The token-aware synchronous gate attempt preserves racing caller cancellation; genuinely contended Stream and TCP sends retain the linked caller/disposal token path. Default-token sends remain allocation-free, and real backpressure cancellation tests cover both transports. |
| Default-profile `PeerRoundTripBenchmarks.MovePlayerAsync` could not produce results because the unary direct-completion path disposed a frame after transferring it to `IRpcFrameChannel`. | `PeerRoundTripBenchmarks.MovePlayerAsync` | Short BenchmarkDotNet run, both low-allocation-profile modes | n/a | n/a | n/a | n/a | n/a | 3,769 (`false`) / 5,180 (`true`) | n/a | 424 (`false`) / 104 (`true`) | Correctness and benchmark-enabling fix only: before, `EndToEndLowAllocationProfile=false` failed with `ObjectDisposedException`; after, both parameter rows completed. No speedup is claimed from this before/after pair. |
| Generated single-stream proxy calls allocated a one-element attachment array. | `StreamedArgumentProxyBenchmarks` | Short BenchmarkDotNet run comparing generated single-stream upload with a two-stream array control | n/a | n/a | n/a | n/a | n/a | 7.556 (`single`) / 17.831 (`two`) | n/a | 40 (`single`) / 120 (`two`) | The generated single-stream path now calls the public `IRpcInvoker` single-attachment overload and the built-in runtime keeps the attachment in `RpcStreamAttachmentSet`; the two-stream control still needs an attachment array. |
