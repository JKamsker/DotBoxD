# DotBoxD Services Benchmark History

All results below are local stopwatch probes on this machine, run in Release mode.

## Ledger

| --- | --- | --- | --- |
| Listener-free RPC request telemetry | this commit | `RpcTelemetryBenchmarks.SuccessfulRequestWithoutListeners` and `PeerRoundTripBenchmarks.MovePlayerAsync` | The focused scope benchmark improved from 55.49 ns / 64 B to 0.54 ns / 0 B by keeping only a one-reference value scope and creating detailed request state when tracing or duration metrics are enabled. Enabled duration tags, activities, and timeout counters remain covered by tests. The end-to-end control corroborated exactly 64 B/request saved in both profiles (640 B to 576 B and 320 B to 256 B); its asynchronous latency moved in opposite directions between runs, so only the isolated latency and end-to-end allocation improvements are claimed. |
| Root RPC proxy cache | `7126981` | `--probe-peer-proxy-cache` | Repeated locked proxy creation measured 33.5 ms / 32,000,040 B for 1,000,000 calls; cached `RpcPeer.Get<IGameService>` measured 36.9 ms / 40 B. This is an allocation-only win for the small root proxy in the probe, with registry-stamp invalidation preserving replacement registration behavior. |
| Owned stream receive tracking | `4ec4439` | `--probe-stream-connection-receive-tracking` | Owned empty-stream receives with legacy active-receive tracking simulated around the current body measured 124.2 ms / 72,001,584 B for 1,000,000 calls; current owned receives measured 113.8 ms / 72,000,040 B. Non-owned streams still track active receives for blocked-read shutdown. |
| Generated Get extension proxy cache | this commit | `--probe-peer-proxy-cache` | Repeated generated `GetGameService` calls used to directly construct `GameServiceProxy`, measuring 8.6 ms / 32,000,040 B for 1,000,000 calls. Routing generated extensions through cached `RpcPeer.Get<IGameService>` measured 37.9 ms / 40 B, so this is an allocation-only tradeoff and preserves shared root proxy identity. |
| Unary frame-channel ownership | this commit | `PeerRoundTripBenchmarks.MovePlayerAsync` | Before this fix, the default profile row failed before measurement with `ObjectDisposedException: PooledBufferWriter` while `EndToEndLowAllocationProfile=true` still completed. After removing the post-transfer dispose, the same short BDN command completed both rows: default profile 3.769 us / 424 B, low-allocation profile 5.180 us / 104 B. This proves the benchmark blocker is gone; it is not a latency-improvement claim. |
| Generated single-stream attachment path | this commit | `StreamedArgumentProxyBenchmarks` | Generated proxies now pass one stream attachment through the `IRpcInvoker` single-stream overload and the runtime stores it in `RpcStreamAttachmentSet` without allocating a one-element attachment array. The short BenchmarkDotNet run measured `SingleStreamUpload` at 7.556 ns / 40 B/op and the two-stream array control `TwoStreamUpload` at 17.831 ns / 120 B/op. The two-stream path still needs an attachment array, so this step claims the single-stream allocation reduction only. |
| Generated zero-parameter metadata arrays | this commit | `--probe-generated-metadata-parameters` | Generated service metadata now emits `Array.Empty<GeneratedParameter>()` for methods with no metadata parameters instead of allocating a zero-length array in the generated static initializer. The 1M metadata-array probe measured the legacy zero-parameter allocation at 24,000,040 B and the current zero-parameter path at 40 B. The one-parameter control still allocates its metadata array. |
| Published RPC peer startup fast path | this commit | `--probe-peer-start-guard` | Ten million repeated healthy `Start()` checks improve from 68.9-71.8 ms (6.9-7.2 ns/op) to 8.2-8.3 ms (0.8 ns/op), with zero allocations before and after. A separate completion publication prevents callers from bypassing a partially completed first start; disposed and remotely closed peers retain the locked exception path. |
| Cached constructor replay metadata and indexed comparison | this commit | `--probe-constructor-replay` | Stable exact/base-declared constructor DTOs fall from 256 to 128 B per serialization. Four-process medians improve from 145.5 to 103.0 ns (-29.2%) and 145.4 to 116.6 ns (-19.8%). The cache stores reflection metadata only: constructor replay and comparison still run for every value, and runtime type mismatches retain dictionary resolution. |
| Lock-free later-deadline timeout scheduling | this commit | `--probe-pending-timeout-scheduler` | Ten million no-op later-deadline schedules improve from 67.2-70.1 ms (6.7-7.0 ns/op) to 13.9-14.8 ms (1.4-1.5 ns/op), with zero allocations before and after. The earliest hint is published with volatile writes, the locked path revalidates before changing the timer, and a paused timeout-scan regression proves a concurrent later deadline is retained. End-to-end finite-unary timings moved directionally but remain process-noisy, so only the isolated scheduler latency is claimed. |
| Frame-scoped receive idle timeout | this commit | `--probe-stream-connection-pending-receive`, `--probe-stream-connection-receive-tracking`, and named-pipe IPC profile | Directly awaiting stream reads inside the existing exact-read state machine removes the two finite-timeout wrapper boxes: 100,000 forced-pending frames fall from 79,200,480 B (792 B/frame) to 47,200,480 B (472 B/frame), exactly matching the infinite-timeout control. Ten alternating process pairs improve the forced-pending finite median 20.4% (1,343.8 to 1,069.1 ns/frame); the paired infinite control is flat (900.5 to 896.6 ns). Keeping one timer armed across prefix and body reduces a typical frame from four timer operations to three; the synchronous finite median improves from 372.1 to 284.5 ns (23.5%) while its infinite median remains flat. Three 20,000-call named-pipe profiles fall from 1,264.2-1,264.3 to 944.2-944.3 B/call for Add and 1,264.0-1,264.1 to 944.0-944.1 B/call for Echo. |
| Direct built-in raw sends | this commit | `--probe-peer-frame-send` | Built-in Stream/named-pipe and TCP channels already validate and serialize every send, so raw peer sends now bypass the redundant peer semaphore through the same internal capability as owned frames. Five TieredCompilation-disabled processes reduce median peer overhead from 44.0 to 12.4 ns for default tokens and from 42.5 to 12.2 ns for reusable live tokens (71.8% / 71.3%), with zero allocations throughout. Closed, pre-canceled, malformed, and mixed frame/raw cases remain enforced by the peer/transport boundary; third-party channels retain peer serialization and cancellable waiters. |
| Terminal frame-timeout source disposal | this commit | `--probe-frame-read-timeout-source-lifetime` | Wrapper disposal now linearizes with only slow source creation/replacement, so a partial read cannot rearm after connection disposal and retain a new timer. Ten alternating process pairs keep the cached Start/Cancel median flat at 81.1 to 79.5 ns and 0 B/op; source allocation remains exactly 40 B. The allocation-free wrapper monitor adds 6.4 ns to one-time Dispose (3.4 to 9.8 ns median) and avoids the 32 B/source cost of a separate gate. Post-disposal Start changes from recreating 144 B unlinked or 192 B linked state to throwing `ObjectDisposedException`; deterministic connection coverage proves no second read or retained source. |
| One-state-machine Stream/TCP frame receive | this commit | `--probe-stream-connection-pending-receive`, `--probe-stream-connection-receive-tracking`, and `--profile-ipc namedpipe 20000 --low-alloc` | Each transport now directly owns one prefix/body receive loop, replacing the outer receive plus two exact-read async boxes with one state machine while sharing synchronous EOF/timeout primitives. A single remaining-byte counter keeps the pending box compact: the forced two-pend lane falls exactly from 472 to 192 B/frame (59.3%) while retaining two reads. Five alternating processes improve finite/infinite medians from 1,097.0/930.9 to 776.3/620.1 ns (29.2%/33.4%). Real named-pipe allocation falls from 472.1 to 336.1 B/frame and TCP from 336.1 to 192.1 B/frame; OS latency remains too noisy to claim. The synchronous control stays at 32 B/frame and improves from 291.5/156.5 to 260.5/120.5 ns (10.6%/23.0%). Full named-pipe unary Add/Echo calls fall by about 272 B each, from 944.2-944.3/944.0-944.1 to 672.2/672.0-672.1 B/call. Direct ownership also preserves the original connection-construction allocation; a composed reader (+56 B/connection) and generic context (3.5-9.2% slower) were measured and rejected. |
| Adaptive TCP/named-pipe receive lookahead | this commit | `--probe-transport-receive-lookahead`, `--probe-transport-large-frame-receive`, `--probe-transport-idle-receive-footprint`, `--probe-stream-connection-pending-receive`, and `DotBoxD.LookaheadCapacityProbe` | Owned named-pipe and TCP connections start with an exact four-byte prefix and use a 16 KiB pooled body window only when that prefix completed synchronously and the frame fits. Five alternating-process medians improve prebuffered 16-frame batches from 737.2 to 384.4 ns/frame for direct named pipes (47.9%) and 688.6 to 286.7 ns/frame for TCP (58.4%). With the first receive genuinely pending before every write, named pipes improve from 2,139.2 to 1,605.2 ns/frame (25.0%) and TCP from 1,864.4 to 1,428.6 ns/frame (23.4%). Counted named-pipe reads fall from 2.000 to 0.125/frame buffered and 0.250/frame pending; allocations remain exactly 32.0/51.0 B per named-pipe frame and 32.0/42.0 B per TCP frame. A missed exact-body lookahead backs off for 255 frames, while real carry stays enabled. One thousand pending idle connections per OS transport retain zero lookahead windows. All four Tiering-disabled batch-1 controls improve 0.4-3.1% with identical allocation; forced-pending exact finite/infinite controls move +0.4%/-1.5% at exactly 192 B/frame. Production 256 KiB medians improve 4.5% named pipe and 3.3% TCP. The standalone raw-reader 32/64 KiB regressions informed the selected bound but do not model the final adaptive path, which reads larger frames directly. |
| Allocation-free terminal TCP send wakeup | this commit | `--probe-tcp-send-gate-contention`, `--probe-transport-connection-construction`, and `--probe-stream-connection-send-gate` | TCP disposal now publishes closure and injects one terminal semaphore permit instead of retaining a per-connection disposal CTS. Waiters acquire, observe closure, and cascade the permit; mixed cancellation/disposal and future sends are pinned by real-backpressure tests. Five-process contended enqueue medians improve from 665.4 to 160.1 ns/waiter with default tokens and from 621.8 to 254.7 ns/waiter with live reusable tokens. Versioning note for the single-stream invoker overloads: the `RpcStreamAttachment` overloads on
`IRpcInvoker` are public generated-proxy ABI so generated proxies can make the allocation-free
single-attachment call across assembly boundaries. Their default interface implementations keep
custom invokers source-compatible by forwarding through the existing array overloads; the built-in
`RpcPeer`/outbound invoker override them for the allocation-free runtime path.
